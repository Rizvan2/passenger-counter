# Согласованность подсчёта: фильтр прохожих, вход по born-at-door, аудит дыр

## 1. Мелкие силуэты на улице больше не считаются (PASSERBY)
Прохожий, видимый сквозь проём, остаётся МЕЛКИМ весь свой трек — он никогда не подходит к
камере. Реальный входящий/выходящий хотя бы раз крупный (у двери).
- В TrajectorySample добавлен bodyHeightRatio (высота ТЕЛЕСНОГО бокса / высота кадра).
- Оффлайн-классификатор: если max(bodyHeightRatio) за весь трек < pc.count-min-body-height-ratio
  (0.18) -> судьба PASSERBY, не считается ни входом, ни выходом, какие бы зоны якорь ни задевал.
- Live-счётчик: такие треки вообще не подаются в стриминговую state-machine (AnalysisService).
Тюнинг: если далёкие входящие у двери вдруг фильтруются — снизь до 0.14; если прохожие
проскакивают — подними до 0.22.

## 2. Вход при рваном id: born-at-door (главная причина недосчёта входящих)
Когда id рвётся посреди входа, «салонная половина» человека рождается уже в двери и не имеет
устойчивого OUTSIDE-визита -> раньше классификатор давал ONBOARD_STATIONARY и вход терялся.
В проекте уже была ветка born-in-door, но БЕЗ ПРОВЕРОК — любой стоящий в салоне, чей якорь
задел полигон двери (а DOOR перекрывает часть салона), получал фантомный +1 вход.
Ужесточено до доказательного прибытия (все три условия сразу):
- контакт с дверью в НАЧАЛЕ трека (первые min-stable-frames сэмплов), а не любое касание;
- голова ВЫРОСЛА от ранних сэмплов к устойчивому салонному визиту (шёл К камере);
- якорь реально сместился (стоящие отфильтровываются).
Ветка срабатывает только при boardings==0 && alightings==0.

## 3. Аудит (что проверено и признано корректным)
- Выход при рваном id: «уличная половина» выходящего = UNDECIDED (не считается), выход даёт
  «салонная половина» через lost-at-exit -> двойного счёта нет.
- Ложный вход от выходящего с рваным id: born-at-door требует РОСТ головы, выходящий
  уменьшается -> не проходит.
- Стоящие/сидящие: не растут и не двигаются -> ONBOARD_STATIONARY, только в начальное заполнение.
- ReID-батч и counted-continuation (внесённые ранее) на месте; конфликт-правок нет.

## Новые параметры (application.yml)
pc.count-min-body-height-ratio: 0.18

## Тесты (TrackFateClassifierTest)
+ far small silhouette -> PASSERBY (0/0)
+ born-at-door с ростом -> 1 вход
+ стоящий, задевший полигон двери -> 0 (фантомный вход отклонён)
Все прежние тесты сохранены. Запуск без моделей: ./gradlew test

## Что осталось главным ограничителем (не код)
FPS ~1.7 на CPU: между кадрами люди прыгают, id рвутся чаще, чем нужно. Батч-ReID уже внесён;
следующий большой рычаг — GPU (onnx-provider: cuda) и process-every-n-frames: 1.

# Довесок 2: фантомные выходы умерших id и слияние двух выходящих

## Фантомный выход стоящего (id умер -> «вышел»)
Причины: (а) ветка lost-at-exit была с условием ИЛИ (tail==OUTSIDE || shrank || moved) — стоящему
хватало дрожания бокса; (б) человек стоит там, где DOOR перекрывает салон -> endAtExit срабатывал.
Исправлено:
- Условие ужесточено до И: moved && (shrank || tail==OUTSIDE). Одиночное «чуть уменьшился» или
  «чуть сдвинулся» больше не выход.
- ДЕТЕКЦИЯ ПРЕЕМНИКА (главное): если сразу после смерти трека A (<= count-successor-max-gap-frames)
  рядом с его последней позицией (<= count-successor-max-distance-px) РОЖДАЕТСЯ новый трек,
  который в первые кадры СТОИТ НА МЕСТЕ — это смена id того же стоящего человека, и lost-at-exit
  для A подавляется. В лог пишется «lost-at-exit suppressed: stationary newborn successor».

## Слияние двух выходящих -> раньше 1 выход, теперь 2
Различитель слияния и смены id: при слиянии второй трек УЖЕ СУЩЕСТВОВАЛ (сосуществовали), при
смене id преемник рождается ПОСЛЕ смерти. Подавление действует только на новорождённых
неподвижных преемников, поэтому:
- A, слившийся в B на пути к двери (двигался + уменьшался из-за окклюзии, умер у выхода) ->
  получает свой lost-at-exit (+1);
- B, доживший до улицы -> обычный переход (+1). Итого 2. Покрыто тестом.

## Новые параметры
pc.count-successor-max-gap-frames: 15   (сырые кадры; при process-every-n=3 это ~5 обработанных)
pc.count-successor-max-distance-px: 120

## Новые тесты
+ стоящий с умершим id и неподвижным преемником -> 0 выходов
+ двое выходящих со слиянием -> 2 выхода

# Довесок 3: поглощение бокса соседом и окно преемника

## «Стоял в салоне, слился с соседом, засчитан выходом»
Когда чужой бокс поглощает стоящего, всё «движение» жертвы живёт в ОДНОМ испорченном финальном
сэмпле (прыжок к соседу + сжатие). Настоящий выходящий покрывает путь за несколько кадров.
Добавлен sustained-guard: предпоследний сэмпл тоже должен быть уже в пути (>= 0.6 * min-movement
от салонной позиции). Прыжок только в финальном кадре -> «box-merge artifact», выход отклонён.
Слияние двух РЕАЛЬНО выходящих не задето: у них путь устойчивый (тест сохранён, 2 выхода).

## Окно преемника расширено: 15 -> 50 сырых кадров
15 кадров (~0.6с) слишком мало: в толпе детектор теряет стоящего на несколько обработанных
кадров, преемник рождается позже окна, подавление молчало. 50 (~2с при 25fps) покрывает
типичный разрыв. Если стоящие всё ещё получают выходы — поднимай до 80-100 (осторожно: слишком
широкое окно может подавить реальный выход, если на то же место кто-то встал).

## Новый тест
+ стоящий, поглощённый боксом проходящего соседа (без новорождённого преемника) -> 0 выходов

## ВАЖНО: какие числа смотреть
Live-панель во время анализа показывает ПОТОКОВЫЙ счётчик. Итоговые числа в offline-режиме
заменяются В КОНЦЕ ролика классификатором (лог «Offline reconciliation: ... streaming(...) ->
offline(...)»). Сравнивай именно конечные числа после завершения анализа. Если фантомный выход
есть в live, но исчез в offline — система работает как задумано.

# Довесок 4: хаос id после перехода на РОДНОЙ детектор голов

## Причина
С detector=head детекция = крошечный головной бокс, body==null -> вся стабилизация «трекинг по
телу» отключилась молча. Трекер сопоставлял головы: IoU между кадрами рвётся (голова за 3 кадра
смещается больше своего размера), ReID резал кроп с головы и давал мусорные эмбеддинги — мусор
случайно проходил порог похожести, отсюда переприсвоения чужим людям, потерянные id и фантомы.

## Исправление 1: синтетическое тело из головы
Detection.syntheticBodyFromHead(): головной бокс расширяется в торс+голову (ширина x2.8,
высота x5.5, центр по голове, верх сохранён). AnalysisService.trackingDetections() прикрепляет
его как body для detector=head. Трекинг и ReID работают по крупному стабильному боксу
(одежда в кропе — то, на чём OSNet обучен), СЧЁТ остаётся по голове.
Параметры: pc.head-body-width-multiplier: 2.8, pc.head-body-height-multiplier: 5.5.
Если тела соседей стали слишком широкими и слипаются в матчинге — уменьшай width к 2.2.

## Исправление 2: гейт рождения треков
Новый id создаётся только от уверенной детекции (pc.track-new-min-confidence: 0.45).
Слабые блобы (блики в проёме, сумки, полузакрытые фрагменты 40-45%) могут МАТЧИТЬСЯ к
существующим трекам и воскрешать их, но не рождают фантомных пассажиров.
На скрине фантомы были именно 41-45%.

## Проверено
isAlighted-треки по-прежнему исключены из всех путей воскрешения — id вышедшего не достаётся
следующему. Тест на синтез тела добавлен (DetectionTest).
