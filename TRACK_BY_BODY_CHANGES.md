# Фикс скачущих id: трекинг по телу, счёт по голове

## Причина бага
head-tracking (enabled=true) заменял person-боксы на маленькие head-регионы ДО трекинга.
PersonTracker сопоставлял людей по крошечным головным боксам: IoU между кадрами проваливался,
порог по расстоянию (завязан на размер бокса) становился мизерным, а ReID (OSNet) на кропе
головы давал слабый эмбеддинг. Итог — id одного человека постоянно менялся, каждый новый id
считался новым выходом; плюс позиционное воскрешение у двери отдавало id следующему человеку.

## Что сделано (минимально инвазивно)
Голова по-прежнему используется для СЧЁТА (зоны/якорь/масштаб не тронуты), но ТРЕКИНГ вернулся
на тело:
- `model/Detection.kt`: добавлено поле `body: Detection?` (полный person-бокс) и `bodyOrSelf`.
  headRegion() теперь возвращает head-бокс с `body = null`.
- `service/AnalysisService.kt` (trackingDetections): к каждому head-боксу прикрепляется исходный
  person-бокс: `headRegion(...).copy(body = person)`. Вниз по конвейеру всё так же идёт head-бокс
  — счётчик, зоны, траектории, emit НЕ менялись.
- `service/PersonTracker.kt`:
  * IoU, расстояние центров и порог расстояния считаются по ТЕЛУ (`bodyOrSelf`);
  * ReID-эмбеддинг снимается с ТЕЛА (`extract(frame, det.bodyOrSelf)`);
  * позиционный fallback (findByLineSideFallback) теперь срабатывает ТОЛЬКО когда эмбеддинга нет
    вообще. Если ReID доступен, но не подтвердил — заводится новый id, а не наследуется чужой
    трек. Это и есть «не передавать id следующему выходящему в том же месте».
  * зон-логика (сторона/дверь) осталась на голове — там важна точка подсчёта, а не тело.

## Эффект
- Один человек = один id: тело большое и стабильное, OSNet на теле даёт нормальный эмбеддинг.
- id вышедшего не переходит следующему, кто встал на то же место у двери.
- Счёт (вход/выход по зонам + масштаб головы) не изменился — работает как раньше, но на
  стабильных id, поэтому серия ложных выходов от скачущих id уходит.

## Совместимость
- detector=head (настоящие головы): body==null, трекинг по голове как раньше.
- head-tracking.enabled=false: трекинг и счёт по телу.

## Проверка
- Юнит-тесты моделей/зон/классификатора запускаются без ONNX: `./gradlew test`.
- На ролике: id одного прохода не меняется (лог Tracker created/resurrected);
  двое выходящих подряд в одном месте -> разные id; один выход на человека.
- Существующий предохранитель `count-duplicate-alighting-distance-px: 85` оставлен как второй
  рубеж, но первопричина (id) теперь чинится в трекере.

# Довесок: почему ReID (память по внешности) не возвращал треки — и что починено

Память по внешности В ПРОЕКТЕ ЕСТЬ — это OSNet (ReidService). Но она включается только если
файл модели `models/osnet_x0_25.onnx` реально загружен. ПЕРВОЕ, что нужно проверить в логах
старта:
- `ReID (OSNet) loaded ...` — память включена;
- `ReID model not available. Re-identification disabled.` — модели нет, никакого запоминания,
  id восстановить нечем (нужно докачать/положить osnet в models/).

Даже когда ReID включён, он раньше промахивался по трём причинам — все исправлены:

1) Искал только в АРХИВНЫХ треках (lostTracks), а трек попадает туда лишь через
   `track-active-lost-frames` (45) кадров после потери. Первые ~45 кадров человек «коастит» в
   активных — и ReID его не проверял. Реаппир со сдвигом → новый id.
   Исправлено: findByReid теперь ищет и в коастящих активных треках (framesSinceUpdate>0), не
   только в архивных.

2) Эмбеддинг снимался ОДИН раз (на создании) и устаревал — первый кадр часто спиной/мелкий/
   перекрыт, и по нему потом никто не находился.
   Исправлено: эмбеддинг обновляется скользящим средним (EMA, blend 0.2) каждые
   `reid-refresh-every-frames` (4) кадра, память остаётся актуальной.

3) Кроп для ReID имел padding под ГОЛОВУ (padBottom = 1.15×высоты — вниз на целый бокс). После
   перехода на кроп по ТЕЛУ это раздувало вырез в пол/соседа.
   Исправлено: padding уменьшен (0.10/0.05/0.05) — аккуратный вырез человека.

Плюс из прошлой правки: позиционный fallback у двери теперь только при отсутствии эмбеддинга,
чтобы id вышедшего не доставался следующему на том же месте.

Новый параметр: `pc.reid-refresh-every-frames: 4`.
Если id всё ещё иногда двоятся — понизь `reid-similarity-threshold` с 0.65 к ~0.55 (осторожно:
слишком низко → риск склеить двух разных людей).
